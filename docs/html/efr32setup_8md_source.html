1 EFR32 Setup {#EFR32_Setup}
2 ============
3 
4 While RAIL attempts to be chip agnostic there are certain hardware differences
5 that can't be overlooked. Where possible, missing features will be simulated in
6 software, but performance characteristics may vary. This section will cover what
7 differs for the EFR32.
8 
9 ## Hardware Initialization
10 
11 The EFR32 comes with versions of EMLIB and EMDRV to create a basic HAL layer.
12 A lot of this initialization code is completely up to you, but there are a
13 couple of requirements when you're building a RAIL app. Specifically, the radio
14 will only work if you are running off a high precision crystal oscillator. Since
15 some APIs will assume this is running you should make sure to initialize and
16 switch to the crystal before calling any radio APIs.
17 
18 For the WSTK you can use the crystal configuration stored in the bspconfig.h
19 header file for your specific kit. Example code for this is shown below. If you
20 have a custom hardware layout you may want to create your own HFXOInit structure
21 to account for things like your specific CTUNE value.
22 
23 ~~~~~~~~~~{.c}
24 #include "bsp.h" // Contains WSTK versions of the HFXO init structure
25 
26 void efrStartup(void)
27 {
28  CMU_HFXOInit_TypeDef hfxoInit = CMU_HFXOINIT_WSTK_DEFAULT;
29 
30  // Initialize the HFXO using the settings from the WSTK bspconfig.h
31  // Note: This configures things like the capacitive tuning CTUNE variable
32  // which can vary based on your hardware design.
33  CMU_HFXOInit(&hfxoInit);
34 
35  // Switch HFCLK to HFXO and disable HFRCO
36  CMU_ClockSelectSet(cmuClock_HF, cmuSelect_HFXO);
37  CMU_OscillatorEnable(cmuOsc_HFRCO, false, false);
38 }
39 ~~~~~~~~~~

41 ## Radio Specific Initialization
42 
43 Currently there are two initialization functions for the radio that depend on
44 your board configuration and must be manually called at startup. You __MUST__
45 initialize the power amplifier (PA) in order to transmit. You __MUST__ also
46 initialize the PA before intializing RAIL for the library to properly account
47 for the PA settings during timing calculations. You may initialize the packet
48 trace (PTI) interface if you want to use this for debugging.
49 
50 ### Power Amplifier (PA) Initialization
51 
52 There are several different PAs available for the EFR32 family of chips. The
53 specific set of PAs you have available and the supported power levels are
54 determined by what part number you're using. To fully understand your part and
55 what options you have please consult the data sheet for your part.
56 
57 Once you have chosen the PA configuration that you need it's important that you
58 initialize the PA before doing any radio operations, but after the HFXO has been
59 enabled and switched to. The structures and APIs for doing this can be found in
60 the pa.h file. An example of initializing the high power 2.4GHz PA is shown
61 below.
62 
63 ~~~~~~~~~~{.c}
64 #include "pa.h"
65 
66 void initPa(void)
67 {
68  RADIO_PAInit_t paInit = {
69  PA_SEL_2P4_HP, // Mode: PA_SEL_SUBGIG, PA_SEL_2P4_LP, PA_SEL_2P4_HP
70  PA_VOLTMODE_DCDC, // VPA mode: PA_VOLTMODE_VBAT, PA_VOLTMODE_DCDC
71  100, // Desired output power in dBm * 10
72  0, // Output power offset in dBm * 10
73  10, // Desired ramp time in us
74  };
75 
76  // Initialize the PA (must be done only after the HFXO is enabled)
77  RADIO_PA_Init(&paInit);
78 }
79 ~~~~~~~~~~

81 ### Packet Trace (PTI) Initialization
82 
83 Packet trace on the EFR32 provides a mechanism for viewing transmitted and
84 received radio packets for network sniffing or debugging. It can also be
85 captured by a WSTK and sent to Simplicity Studio for viewing the data in its
86 Network Analyzer tool.
87 
88 If you want to use this functionality, you must configure what pins you want to
89 use for this output and, optionally how you want to format the data. Note that
90 the WSTK requires the following output format:
91 
92  Mode: 8 bit UART mode
93  Baudrate: 1.6Mbps
94  Framing Signal: Enabled
95 
96 If you change anything here it is currently unsupported by the WSTK. In the
97 future we may include support for alternate output modes.
98 
99 To choose the pins you want you must look in the data sheet for your part and
100 select a valid route location for the FRC_DOUT and FRC_DFRAME signals. These map
101 to PTI.DATA and PTI.FRAME on the WSTK respectively. Once you've found pins that
102 work for your hardware you need to configure the RouteLocation, port, and pin
103 fields in the PTI initialization structure. When using the WSTK for example,
104 you should initialize things as follows.
105 
106 ~~~~~~~~~~{.c}
107 #include "pti.h"
108 
109 void initPti(void)
110 {
111  RADIO_PTIInit_t ptiInit = {
112  RADIO_PTI_MODE_UART, // Only supported output mode for the WSTK
113  1600000, // Choose 1.6 Mbps for the WSTK
114  6, // WSTK uses location 6 for DOUT
115  gpioPortB, // FRC_DOUT#6 is PB12
116  12, // FRC_DOUT#6 is PB12
117  6, // UNUSED IN UART MODE
118  gpioPortB, // UNUSED IN UART MODE
119  11, // UNUSED IN UART MODE
120  6, // WSTK uses location 6 for DFRAME
121  gpioPortB, // FRC_DOUT#6 is PB13
122  13, // FRC_DOUT#6 is PB13
123  };
124 
125  // Initialize the Packet Trace Interface (PTI)
126  RADIO_PTI_Init(&ptiInit);
127 }
128 ~~~~~~~~~~

130 ### Other Radio GPIO Functions
131 
132 There are various useful signals related to the radio that can be output on a
133 GPIO using the Peripheral Reflex System (PRS). The PRS is an advanced system
134 where you can route signals to channels and then output those channels to a
135 number of configurable locations. There's a lot more to it than that, so
136 consult the reference manual chapter on the PRS if you want to understand it
137 better.
138 
139 A list of some of the most interesting PRS signals related to the radio is shown
140 below along with how to enable them in the PRS.
141 
142 | Signal | Summary | PRS Source | PRS Signal |
143 |:----------------|:----------------------|:-----------|:-----------|
144 | RACACTIVE | Radio enabled | 0x20 | 0x00 |
145 | RACTX | Transmit mode enabled | 0x20 | 0x01 |
146 | RACRX | Receive mode enabled | 0x20 | 0x02 |
147 | RACLNAEN | LNA enabled for Rx | 0x20 | 0x03 |
148 | RACPAEN | PA enabled for Tx | 0x20 | 0x04 |
149 | MODEM_FRAMEDET | Frame detected | 0x26 | 0x00 |
150 | MODEM_PREDET | Preamble detected | 0x26 | 0x01 |
151 | MODEM_TIMDET | Timing detected | 0x26 | 0x02 |
152 | MODEM_FRAMESENT | Frame sent | 0x26 | 0x03 |
153 | MODEM_SYNCSENT | Syncword sent | 0x26 | 0x04 |
154 | MODEM_PRESENT | Preamble sent | 0x26 | 0x05 |
155 
156 The example below shows you how to configure a PRS channel to output RACRX on
157 a GPIO. For this example we assume you're using the SLWSTK6001A_EFR32MG kit. The
158 specified configuration will put RACRX on PRS Channel 0 and output PRS Channel 0
159 on the EFR32's PC10 which is wired to WSTK_P12 and EXP_HEADER15 on the WSTK.
160 
161 ~~~~~~~~~{.c}
162 #include "em_cmu.h"
163 #include "em_gpio.h"
164 #include "em_device.h"
165 
166 void enableDebugGpios(void)
167 {
168  // Turn on the PRS and GPIO clocks so we can access their registers
169  CMU_ClockEnable(cmuClock_PRS, true);
170  CMU_ClockEnable(cmuClock_GPIO, true);
171 
172  // Configure PC10 as an output
173  GPIO_PinModeSet(gpioPortC, 10, gpioModePushPull, 0);
174 
175  // Configure PRS Channel 0 to output RACRX
176  PRS->CH[0].CTRL = (0x20 << _PRS_CH_CTRL_SOURCESEL_SHIFT)
177  | (0x02 << _PRS_CH_CTRL_SIGSEL_SHIFT)
178  | PRS_CH_CTRL_EDSEL_OFF;
179 
180  // Configure PRS Channel 0 to use output location 12 (PC10 - see datasheet)
181  PRS->ROUTELOC0 &= ~_PRS_ROUTELOC0_CH0LOC_MASK;
182  PRS->ROUTELOC0 |= PRS_ROUTELOC0_CH0LOC_LOC12;
183 
184  // Enable PRS Channel 0
185  PRS->ROUTEPEN |= PRS_ROUTEPEN_CH0PEN;
186 }
187 ~~~~~~~~~

189 ## Required Dependencies
190 
191 Most of the RAIL library is self contained, however, there are some dependencies
192 on external functions. Specifically, we rely on functions from the C standard
193 library, EMLIB, and CMSIS. Below is a complete list of these dependencies. If
194 you want to change the implementation of any of these functions while
195 maintaining the functionality that's fine, just know that they could impact RAIL
196 operation.
197 
198 | Group | Functions |
199 |-------|-----------|
200 | CMSIS | `SystemHFXOClockGet()`<br>`SystemLFRCOClockGet()`<br>`SystemLFXOClockGet()`<br>`SystemULFRCOClockGet()`|
201 | EMLIB (em_int) | `INT_Disable()`<br>`INT_Enable()` |
202 | EMLIB (em_cmu) | `CMU_ClockFreqGet()`<br>`CMU_ClockEnable()`<br>`CMU_ClockSelectSet()`<br>`CMU_OscillatorEnable()` |
203 | EMLIB (em_gpio) | `GPIO_PinModeSet()` |
204 | EMLIB (em_system) | `SYSTEM_ChipRevisionGet()` |
205 | stdlib | `memcpy()`<br>`memset()` |

207 ## Peripherals Consumed by RAIL
208 
209 For now, RAIL consumes only peripherals that are a part of the radio. In the
210 future we will likely need to use a few of the customer facing peripherals for
211 enhanced functionality. To prepare you for this we're listing peripherals likely
212 to be shared with RAIL in the future.
213 
214 * PRS Channels
215  * RAIL will likely need one or two PRS channels to perform various
216  synchronization and signaling. We will leave the vast majority of these
217  channels alone so that they can be used by customers.
218 * RTCC Timer
219  * While RAIL won't require the whole RTCC timer we will need to use one
220  channel of it to synchronize our internal timebase across deep sleep.
221  * This is required because the RTCC timer should be one of the only timers to
222  run throughout deep sleep and we don't want the RAIL timebase to slip every
223  time we go to sleep.
224 
225 ## Entropy Generation
226 
227 The EFR32 supports true entropy collection using the radio. It is able to
228 collect 1 bit per radio clock cycle while in receive mode. This means that we
229 must enable the receiver when collecting entropy and if you attempt to transmit
230 or otherwise delay entry into receive it will take us a longer time to collect
231 the data. You can still receive packets while collecting entropy and this does
232 mean that requesting random numbers longer than the length of your preamble and
233 sync word may trigger a packet reception. This may be fixed in a future revision
234 but for now this is done to ensure proper packet reception during random data
235 collection.
236 
237 Due to the nature of random data collection we will always wait for the full
238 amount requested so we should return either the number of bytes requested or
239 zero. Zero is returned if the radio is uninitialized and cannot be enabled for
240 collection.

242 ## Radio Calibration
243 
244 The EFR32 supports two radio calibration options: image rejection (IR)
245 calibration and VCO temperature calibration. You may choose to enable one or both
246 of these based upon your specific situation. We always recommend using both, but
247 here is more information.
248 
249 * Image Rejection Calibration (IRCAL)
250  * This calibration should be run once every time your PHY config changes. It
251  is based on the modulation scheme, frequency band, and other radio settings.
252  RAIL will request this via the RAILCb_CalNeeded() callback with the
253  \ref RAIL_CAL_ONETIME_IRCAL bit set in RAIL_CalPendingGet().
254  * Using a proper value for this will improve sensitivity by several dBm, so
255  it's highly recommended.
256  * It can take on the order of 700 ms to complete this calibration.
257  You may want to save off a known good value for this calibration and load
258  it each time you switch PHYs to save time.
259  * This calibration should be initialized before channel configuration in order
260  to provide parameters specific to the current radio config. The
261  initialization parameters come from Simplicity Studio's application builder.
262  * You cannot use the radio while this calibration is being performed or you may
263  generate an incorrect calibration. Application code should ensure that the
264  radio remains in the idle state during this calibration.
265  * This calibration should be performed only after the radio config has been
266  downloaded to the radio, calibration initialization has occured, and the
267  frequency channel has been configured.
268 * VCO Temperature Calibration (VCO_TEMPCAL)
269  * If you sit in receive for a very long time and experience temperature swings
270  it's possible for the radio to drift off frequency. When this situation is
271  detected RAIL will request this calibration via the RAILCb_CalNeeded()
272  callback with the \ref RAIL_CAL_TEMP_VCO bit set in RAIL_CalPendingGet().
273  * This calibration is automatically run every time we enter receive. So if you
274  are frequently forcing a re-enter into receive mode you may not need to
275  enable this calibration.
276  * We always recommend that you enable and handle this calibration since it
277  doesn't add much overhead and is much safer if you ever do end up in this
278  situation.
